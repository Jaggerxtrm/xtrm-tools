## Code Changes

### Security — allow-list hardening (xtrm-86z0)

`config/settings.json`: replaced `Bash(bd:*)` and `Bash(git:*)` wildcards with explicit safe subcommand allow-lists. Safe `bd`: show, list, ready, stats, search, prime, memories, recall. Safe `git`: status, log, diff, show, fetch. Mutating commands (bd close/update/create, git commit/push/rebase/reset) excluded — these now trigger hooks normally.

`cli/src/core/sync-executor.ts`: added `normalizeLegacyBashAllowlist()` — strips legacy wildcard entries from existing `~/.claude/settings.json` and injects the explicit safe list on next `xtrm install` sync. Handles all existing installs that were previously bypass-prone.

`hooks/beads-memory-gate.mjs`: Stop hook hardened with 3-level issue resolution fallback: (1) `claimed:<sessionId>` KV, (2) `closed-this-session:<sessionId>` KV, (3) branch name inference. No longer relies solely on PostToolUse markers that the allow-list bypass can skip.

### Skills registry — Phase 0 install (xtrm-s3pf)

`cli/src/commands/install.ts`: during `xtrm install`, creates `.xtrm/skills/default` as a symlink to the npm package `skills/` dir, then creates `.agents/skills` as a symlink to `.xtrm/skills/default`. Both fall back to copy if symlink creation fails (Windows / permission-restricted environments).

### Documentation

`docs/skills-registry-exploration.md` (485 lines total across 2 commits): full exploration spec covering architecture analysis, Options A+B with tradeoffs table, compatibility matrix (Pi / Claude / global / local / worktree), `xt skills` CLI contract with `--global/--local` flags, migration plan with idempotency guarantees and rollback, formal three-tier packaging schema (default/extra/user), conflict resolution algorithm, risks and open questions, and Option B phased recommendation.
